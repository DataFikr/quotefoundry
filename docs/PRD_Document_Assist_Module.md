# QuoteFoundry — Document Assist Module (PRD)

## Optional RFQ Document Pre-Fill — Tiered by Cost & Reliability (Add-On to the Quoting MVP)

Version: 3.0 (tiered model: spreadsheet priority + best-effort text-PDF)
Status: Scoped add-on specification
Audience: Claude Code / Cursor / Copilot / AI coding agents
Depends on: QuoteFoundry quoting MVP (manual quote editor, customer list, quote engine, services layer)

---

# 1. Purpose

The estimator's most repetitive administrative task is **retyping the customer's RFQ** — copying part number, material, quantity, and due date from a document into the quote. Document Assist removes that retyping where it can be done reliably and cheaply, while leaving the estimator in full control of every value.

This module is **optional and additive**. It does not change the manual quoting flow. A shop that never uploads a document gets exactly the product they have today.

---

# 2. The Core Design Decision: a Cost-and-Reliability Tier System

Not all documents are equal. Some parse for free and reliably; some cost money and are unreliable. The module is built around a strict tier system that captures the cheap, reliable wins and refuses the expensive, unreliable ones.

| Tier | Input | Method | AI tokens? | Reliability | Behavior |
|------|-------|--------|-----------|-------------|----------|
| **1 — Priority** | XLSX, CSV | Deterministic header matching | **None** | High | Parse & pre-fill |
| **2 — Best-effort** | Text-layer PDF | Embedded-text extraction (+ optional light structuring) | **None to minimal** | Medium | Parse & pre-fill, low-confidence by default |
| **3 — Fallback** | Scanned/image PDF, DWG, STP, DXF, images, ZIP | None — store only | **None** (no OCR) | n/a | Attach file, manual entry |

The line that controls cost is the **text-layer gate** (Section 6): a PDF is only parsed if it already contains selectable text. If it doesn't, it drops to Tier 3 — we never spend vision/OCR tokens trying to read pixels.

---

# 3. Why This Tiering (the reasoning)

* **Spreadsheets are free and reliable.** Structured rows/columns parse deterministically with a library — no AI, no token cost, trustworthy on day one. This is the priority path.
* **Text-layer PDFs are nearly free.** When a PDF was generated by software, its text is embedded and selectable; a library extracts it at no token cost. Turning that raw text into mapped fields *may* use a small AI pass, but the input is text (cheap), not images.
* **Scanned/image PDFs and drawings are expensive and unreliable.** Reading them needs OCR or a vision model — real token cost — and the results are inconsistent because layouts vary wildly. These are explicitly refused (store-only), avoiding both the cost and the false-confidence trap.
* **Drawing files (DWG/STP/DXF) are stored, never parsed.** Geometry parsing is out of scope entirely; the customer's drawing is attached for reference.

This directly addresses the real-world friction: customers sending drawing files shouldn't be forced to also fill a separate spreadsheet. We parse what they send when we can (spreadsheet or text-PDF), store what we can't, and never block the estimator.

---

# 4. Guiding Principle

> Parse the cheap, reliable formats. Refuse the expensive, unreliable ones. Never block the estimator, never spend tokens on pixels.

Document Assist pre-fills a draft. The estimator reviews and corrects every field, exactly as they would when typing from scratch — just faster.

# 6. The Text-Layer Gate (the cost control)

Before parsing any PDF, detect whether it contains an extractable text layer:

* Use a PDF text-extraction library (e.g. `pdf-parse`, `pdfjs`). If it returns meaningful selectable text -> **Tier 2, parse**.
* If it returns little or no text (a scanned image or image-only export) -> **Tier 3, store only**. Do **not** invoke OCR or a vision model.

This single gate is what guarantees the module never incurs vision/OCR token cost. Scanned documents are common in this industry; refusing them at the gate is deliberate and keeps per-quote cost near zero.

---

# 7. Extraction Target Fields

Map parsed values onto the quote editor's existing inputs and a small metadata block. Only fields the editor already uses, plus reference metadata.

Pre-fillable fields:

* Customer PO / reference number
* Part number
* Drawing number (reference only)
* Description / job name
* Material spec
* Material grade
* Thickness
* Quantity
* Finish spec
* Due date / lead time
* Notes / special instructions

Each pre-filled field carries an editable value, a confidence indicator, and a "review this" cue on medium/low confidence.

Fields the engine prices (labor hours, rates) are **never** extracted — those come from the estimator's judgment and the shop's rate library, unchanged.

---

# 8. Parsing Behavior by Tier

### Tier 1 — Spreadsheet (XLSX, CSV) — priority, zero tokens

* Server-side parse with a standard library (e.g. SheetJS).
* **Header matching:** map columns to fields against a synonym list (e.g. "Qty"/"Quantity"/"Order Qty" -> quantity; "Matl"/"Material"/"Material Spec" -> material). Case- and whitespace-insensitive.
* **Confidence by match quality:** exact header = high; synonym/fuzzy = medium; positional guess = low; no match = blank.
* Multi-row sheets: pre-fill the first line item; surface additional rows as draft lines the estimator can accept or discard. Keep simple — one primary item applied, the rest visible.
* No AI call required for structured sheets.

### Tier 2 — Text-layer PDF — best-effort, minimal tokens

* Only entered if the text-layer gate (Section 6) passed.
* Extract the embedded text with the library — **no token cost** for the extraction itself.
* Structuring the raw text into fields: start with **keyword/regex matching** (e.g. find "Qty", "Material", "Due" labels) — also zero tokens. Add a **small AI structuring pass only if** keyword matching proves insufficient on real files; the input is extracted text (a few thousand tokens at most), so cost per quote stays minimal.
* **Default all Tier-2 fields to low/medium confidence** and flag them for review more aggressively than spreadsheets, because PDF layouts vary. Treat Tier 2 as a helpful head-start, not a trusted source.

### Tier 3 — Store-and-fallback — no parsing, no tokens

* Scanned/image PDFs (failed the gate), DWG/STP/DXF, images, ZIP, DOCX: attach to the quote, no parsing.
* UI message: "Attached — couldn't auto-read this one, enter details manually." No error, no dead end.

All tiers: every value is a **draft**; the estimator's review is mandatory; missing fields are left blank, never guessed.

---

# 9. Confidence & Review UX

* High (Tier 1 exact match): pre-filled, normal styling.
* Medium / low (Tier 1 fuzzy, all of Tier 2): pre-filled but flagged for review (subtle marker).
* Missing: blank field, optional "not found in document" hint.
* The estimator edits inline in the existing editor. Correcting a value *is* the approval — no separate gate, consistent with the manual flow.
* **Always show the source tier** subtly (e.g. "from spreadsheet" vs "from PDF — please verify") so the estimator calibrates trust appropriately.

---

# 10. Data & Storage

* Store every uploaded file against the quote (Supabase Storage), scoped by `shop_id` under the same Row-Level Security as everything else.
* Persist: file reference, tier used (parsed-spreadsheet / parsed-text-pdf / stored-only), and the extracted field set (for audit / "where did this value come from").
* No new top-level entity. Files and extraction attach to the existing quote record.

---

# 11. Explicitly Deferred (NOT in this module)

| Deferred | Why deferred |
|----------|--------------|
| OCR / vision parsing of scanned PDFs & images | Real token cost + unreliable; the text-layer gate refuses these by design |
| CAD / DXF / STEP geometry parsing | Heavy; store-only is sufficient until customers pull for it |
| RFQ as a first-class entity | Large data-model and workflow restructure; pre-fill needs none of it |
| Draft BOM builder | A sub-product of its own; not required to pre-fill a quote |
| Quote revisions / customer portal | Post-launch roadmap, unrelated to document pre-fill |
| Heavy AI structuring of messy PDFs | Keyword matching first; expand only if real files justify the cost |

When real customers send documents these don't cover, *that* is the signal to build the next slice — against reality, not assumption.

---

# 12. Acceptance Criteria

This module is complete when:

- The quote editor offers an optional "upload RFQ file" action accepting multiple files
- **Tier 1:** uploading XLSX/CSV pre-fills fields via header matching, at high confidence on exact matches, with zero AI token cost
- **Tier 2:** uploading a text-layer PDF extracts embedded text and pre-fills fields at low/medium confidence, flagged for review
- **Text-layer gate:** a scanned/image PDF (no text layer) is stored only — never sent to OCR or a vision model
- **Tier 3:** uploading any non-parseable file (scanned PDF, DWG, STP, image, ZIP) attaches it and falls back to manual entry with a clear message — no error, no dead end
- Each pre-filled field shows an editable value, a confidence indicator, and its source tier
- The estimator can edit every value before saving
- The manual quoting path is completely unchanged for shops that don't upload
- Files and extracted data are stored against the quote under existing shop isolation

# 13. Build Estimate & Sequencing

**Estimate (solo founder):**

* Tier 1 (spreadsheet) alone: **~3-5 days.** Deterministic, low risk — ship this first.
* Tier 2 (text-layer PDF, keyword-based): **+2-4 days.** The text-layer gate and extraction are quick; the variable is how much structuring the real PDFs need.
* Adding an AI structuring pass for Tier 2, only if needed: **+1-2 days.**

**Recommended sequencing:**

1. Ship the manual quoting MVP.
2. Add **Tier 1 (spreadsheet)** as the first document-assist feature — cheapest, most reliable, zero token cost.
3. Add **Tier 2 (text-layer PDF, keyword-first)** once real customer documents show it's worth it and reveal the common layouts. Start with zero-token keyword matching; add a small AI pass only if accuracy demands it.
4. Keep Tier 3 (store-and-fallback) from the start so no upload ever dead-ends.

**Scope-creep guardrail:** if a task in flight requires OCR/vision on scanned files, CAD geometry parsing, a new RFQ entity, or BOM editing, it has left this module's scope. Stop and defer it to Section 11. The text-layer gate is the bright line — never cross it into pixel parsing for the MVP.

---

# 14. Relationship to the Original RFQ PRD

The original RFQ workflow PRD (v2.0) proposed a full AI document platform — RFQ entity, OCR of drawings, draft BOM, multi-tab review, revisions. Taken whole, that roughly triples the MVP and front-loads the least-certain, most expensive work (drawing OCR) before customer validation.

This module extracts the **cheap, reliable core of that value** — kill the retyping for spreadsheets (free) and text-layer PDFs (near-free) — while refusing the expensive, unreliable parsing at a clear technical boundary (the text-layer gate). It's an optional add-on that never blocks launch, never burdens shops who don't use it, and never spends tokens on documents it can't read reliably. OCR, CAD parsing, and the rest of the original PRD become the staged roadmap in Section 11, built only when real customer documents justify them.
